1. Explicación del Concepto

El Hardening es el proceso de asegurar un sistema reduciendo su superficie de ataque eliminando funciones, puertos, permisos o servicios innecesarios. Se fundamenta en el Modelo de Responsabilidad Compartida, donde el proveedor asegura la infraestructura física (la "nube"), pero el cliente es responsable de asegurar los datos, las identidades y la configuración de los recursos (lo que está "en la nube").

4. Relación con Normativas

Las configuraciones seguras se alinean con estos estándares globales:

  • CIS Benchmarks La guía de oro. Pasos exactos para el endurecimiento de configuraciones en AWS, Azure y GCP.
  • NIST SP 800-53 Definen los controles de gestión de configuración (CM) y control de acceso (AC).
  • ISO/IEC 27017 Aborda el aislamiento de datos y gestión de activos virtualizados.

2. Cómo se Implementa en Cloud (Capas Críticas)

🛡️ Capa de Almacenamiento
  • AWS: S3 Block Public Access a nivel de cuenta.
  • Azure: Deshabilitar acceso público y usar Tokens SAS estrictos.
  • GCP: Habilitar UBLA para evitar ACLs antiguas.
🔑 Capa de Identidad (IAM)
  • Implementar Principio de Mínimo Privilegio (PoLP).
  • Reemplazar Access Keys permanentes por credenciales temporales.
🌐 Capa de Red
  • Cierre de puertos con Security Groups / NSG apuntando a IPs específicas.
  • Asegurar que SSH/RDP nunca estén abiertos a Internet.
# Implementación de S3 Block Public Access a nivel de cuenta
aws s3control put-public-access-block --account-id <tu-id-cuenta> \
  --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true"
# Deshabilitar el acceso público a Blob Storage
az storage account update --name "mystorageaccount" \
  --resource-group myResourceGroup --allow-blob-public-access false

6. Gobernanza Proactiva y Automatización

Las configuraciones manuales fallan por naturaleza humana. La automatización es la única solución escalable frente a la fatiga de configuración:

⚡ IaC y Shift-Left

Utilizar herramientas como Terraform e integrar escáneres en el pipeline M para vulnerabilidades antes del despliegue.

🎯 Validación de Comportamiento

Uso de sondas para verificar si una alerta es explotable, reduciendo falsos positivos en un 93%.

👀 Monitoreo y Auditoría

Vigilancia con AWS CloudTrail o Azure Monitor para detectar quién alteró configuraciones críticas.

🔄 Drift Detection

Alerta inmediata ante cambios manuales (ClickOps) para forzar autoremediación.

🛡️ Patch Management

Hardening del OS en instancias virtuales para cerrar vulnerabilidades críticas de red.

🌐 Auditoría de DNS

Verificación constante de registros DNS CNAME evitando secuestros por subdominios inactivos.

Glosario de Seguridad Cloud

Definiciones críticas requeridas para entender la arquitectura y prevención de brechas.

1. Hardening

Proceso estricto de reducir la superficie de ataque eliminando configuraciones y puertos innecesarios.

2. Superficie de Ataque

Conjunto de todas las vulnerabilidades expuestas. A menor superficie, menor probabilidad de brecha exitosa.

3. PoLP

Principle of Least Privilege. Dar a usuarios e identidades solo los permisos mínimos indispensables.

4. Modelo Resp. Compartida

El proveedor asegura el datacenter físico; el cliente asegura los datos, accesos y configuraciones.

5. Zero Trust

Arquitectura que asume riesgo permanente: "Nunca confíes, siempre verifica" en cada petición interna.

6. Shadow IT

Sistemas levantados sin control ni auditoría por desarrolladores, fuera del rango del firewall SOC.

7. Configuration Drift

Desviación gradual y peligrosa entre la infraestructura actual y su diseño seguro original.

8. Buckets / Blobs

Contenedores de almacenamiento masivo. El principal vector de fuga si se dejan como "Público".

9. Endpoints API

Puertas de entrada digitales. Deben estar cifradas y enmascaradas detrás de gateways o WAFs.

10. IAM

Identity & Access Management. El nuevo perímetro de seguridad donde se decide quién entra.

11. Managed Identities

Identidad sin contraseña otorgada a las máquinas, encriptando su acceso internamente.

12. Bucket Policies (AWS)

Reglas JSON anexadas a S3 que dictan controles absolutos sobre quién puede interactuar con los datos.

13. Block Public Access

Interruptor maestro en la cuenta (AWS/Azure) que impide exponer recursos al internet global.

14. UBLA (GCP)

Uniform Bucket-Level Access. Evita Listas de Control de Acceso obsoletas priorizando IAM central.

15. CSPM

Cloud Security Posture Management. Radares que escanean la nube detectando fallas automáticamente.

16. CNAPP

Plataformas que unifican defensa perimetral, código y monitor de cargas (Todo en Uno).

17. MFA

Multi-Factor Authentication. Evitó miles de ataques corporativos, requiere 2+ métodos de validación.

18. Security Groups

Firewalls de instancia virtual (Stateful). Solo lo que permitas explícitamente atravesará.

19. NSG

Network Security Groups (Azure). Filtros de red aplicados por subred o tarjetas de interfaz.

20. RBAC

Asignar permisos basados estrictamente en el "Rol" y responsabilidades del usuario, no en la persona.

21. Tokens SAS

Firmas temporales en Azure. Si se generan sin fecha de vencimiento y amplios permisos, son letales.

22. Movimiento Lateral

Proceso donde un hacker se esparce desde una máquina débil hacia la base de datos central.

23. Exfiltración

Extracción encriptada o masiva de data hacia servidores bajo control atacante u oscuros.

24. Cryptojacking

Secuestro silencioso de poder de CPU en la nube empresarial para minar criptomonedas.

25. ClickOps

Práctica obsoleta de configurar la nube con clicks manuales, propenso a errores humanos.

26. IaC

Infraestructura como Código (Terraform). Levantar la nube usando un script 100% auditable.

27. Shift-Left

Empujar la seguridad "a la izquierda" del ciclo de software, auditando en el momento de crear el código.

28. ACL

Access Control List. Forma antigua y riesgosa de dar permisos directamente ligados al objeto.
