1. Explicación del Concepto

El Hardening es el proceso de asegurar un sistema reduciendo su superficie de ataque eliminando funciones, puertos, permisos o servicios innecesarios. Se fundamenta en el Modelo de Responsabilidad Compartida, donde el proveedor asegura la infraestructura física (la "nube"), pero el cliente es responsable de asegurar los datos, las identidades y la configuración de los recursos (lo que está "en la nube").

4. Relación con Normativas

Las configuraciones seguras se alinean con estos estándares globales:

  • CIS Benchmarks La guía de oro. Pasos exactos para el endurecimiento de configuraciones en AWS, Azure y GCP.
  • NIST SP 800-53 Definen los controles de gestión de configuración (CM) y control de acceso (AC).
  • ISO/IEC 27017 Aborda el aislamiento de datos y gestión de activos virtualizados.

2. Cómo se Implementa en Cloud (Capas Críticas)

🛡️ Capa de Almacenamiento
  • AWS: S3 Block Public Access a nivel de cuenta.
  • Azure: Deshabilitar acceso público y usar Tokens SAS estrictos.
  • GCP: Habilitar UBLA para evitar ACLs antiguas.
🔑 Capa de Identidad (IAM)
  • Implementar Principio de Mínimo Privilegio (PoLP).
  • Reemplazar Access Keys permanentes por credenciales temporales.
🌐 Capa de Red
  • Cierre de puertos con Security Groups / NSG apuntando a IPs específicas.
  • Asegurar que SSH/RDP nunca estén abiertos a Internet.
# Implementación de S3 Block Public Access a nivel de cuenta
aws s3control put-public-access-block --account-id <tu-id-cuenta> \
  --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true"
# Deshabilitar el acceso público a Blob Storage
az storage account update --name "mystorageaccount" \
  --resource-group myResourceGroup --allow-blob-public-access false

6. Gobernanza Proactiva y Automatización

Las configuraciones manuales fallan por naturaleza humana. La automatización es la única solución escalable frente a la fatiga de configuración:

⚡ IaC y Shift-Left

Utilizar herramientas como Terraform e integrar escáneres en el pipeline M para vulnerabilidades antes del despliegue.

🎯 Validación de Comportamiento

Uso de sondas para verificar si una alerta es explotable, reduciendo falsos positivos en un 93%.

👀 Monitoreo y Auditoría

Vigilancia con AWS CloudTrail o Azure Monitor para detectar quién alteró configuraciones críticas.

🔄 Drift Detection

Alerta inmediata ante cambios manuales (ClickOps) para forzar autoremediación.

🛡️ Patch Management

Hardening del OS en instancias virtuales para cerrar vulnerabilidades críticas de red.

🌐 Auditoría de DNS

Verificación constante de registros DNS CNAME evitando secuestros por subdominios inactivos.

Glosario de Seguridad Cloud

Definiciones críticas requeridas para entender la arquitectura y prevención de brechas.

1. Hardening

Proceso de reducir la superficie de ataque de un sistema eliminando configuraciones innecesarias, aplicando restricciones y siguiendo guías de mejores prácticas de seguridad.

2. Superficie de ataque

Conjunto de todos los puntos de entrada por los que un atacante podría intentar acceder a un sistema. Más configuraciones abiertas = mayor superficie de ataque.

3. Principio de menor privilegio

Regla de seguridad que indica que cada usuario, proceso o sistema debe tener solo los permisos mínimos necesarios para realizar su tarea, y nada más.

4. Buckets/Blobs

Es el contenedor lógico básico de almacenamiento en la nube (ej. AWS S3). Buckets Públicos: Riesgo muy común de exponer datos sensibles por no cambiar la visibilidad a "privada". Brechas: Ocurren por mala configuración del contenedor, permitiendo a quien sea ver o descargar archivos sin contraseñas.

5. Shadow IT

Servicios levantados por desarrolladores sin pasar por el proceso de hardening (ej. bases de datos sin contraseña).

6. Movimiento Lateral

Si una instancia comprometida no tiene hardening de red, el atacante puede saltar a otros servidores críticos.

7. Endpoints

Cualquier dispositivo o punto de conexión que se comunica con una red. En Cloud piensa en un endpoint como una "puerta de entrada" a tus servicios. Si está "expuesto", cualquiera en Internet podría intentar "tocar la puerta".

8. CSPM

Cloud Security Posture Management: Categoría de herramientas de seguridad que escanean continuamente tu infraestructura en la nube para detectar malas configuraciones y vulnerabilidades.

9. CNAPP

Cloud-Native Application Protection Platforms: Evolución del CSPM que unifica múltiples herramientas de seguridad para proteger todo el ciclo de vida de una aplicación en la nube.

10. MFA

Multi-Factor Authentication: Seguridad que requiere que el usuario proporcione dos o más pruebas (factores) diferentes para verificar su identidad antes de acceder.

11. Modelo Resp. Compartida

Define quién es responsable de qué, dividiendo el trabajo entre el proveedor de la nube (el dueño del edificio) y tú (el inquilino).

12. Bucket Policies (AWS)

Documentos JSON que se adjuntan a un contenedor completo en AWS S3. En lugar de dar permisos archivo por archivo, definen reglas a nivel general.

13. Block Public Access

Interruptor de máxima prioridad en AWS. Bloquea instantáneamente cualquier acceso público, ignorando y anulando cualquier política o ACL previa.

14. Blob Storage (Azure)

Servicio de almacenamiento de objetos de Azure. Diseñado para almacenar cantidades masivas de datos no estructurados, como texto, imágenes o videos.

15. allow-blob-public-access false

Configuración específica en Azure que desactiva el acceso anónimo y público a todos los contenedores. Equivalente al Block Public Access.

16. SAS (Azure)

Shared Access Signatures: Token o URL firmada criptográficamente para otorgar acceso temporal y limitado a un recurso específico sin exponer la contraseña principal.

17. UBLA (GCP)

Uniform Bucket-Level Access: Unifica el control de acceso. Al activarla, deshabilita las antiguas ACLs y fuerza a que toda seguridad se gestione mediante IAM.

18. (ACL) Access Control List

Método de seguridad heredado. Lista adjunta a un archivo que dice quién tiene permisos. Es propensa a errores humanos a escala Cloud.

19. Managed Identities

Identidades Administradas (Azure): Da a una aplicación su propia identidad para conectarse a bases de datos sin guardar contraseñas en el código.

20. Security Groups (AWS)

Actúan como firewall virtual a nivel de la instancia. Son stateful: si aprueban la entrada de un dato, aprueban su salida.

21. NSG (Azure)

Network Security Groups: Asociables a subredes completas o interfaces. Filtran el tráfico determinando qué IPs y puertos pueden comunicarse.

22. SSH (Puerto 22)

Secure Shell. Si este protocolo para servidores Linux se deja abierto a todo Internet (0.0.0.0/0), atacantes buscarán tomar control por fuerza bruta.

23. RDP (Puerto 3389)

Remote Desktop Protocol de Windows. Si está expuesto públicamente, es uno de los principales vectores de entrada para ataques de Ransomware.

24. Exfiltración

Transferencia o robo no autorizado de datos desde un sistema hacia el exterior (robando información confidencial masivamente).

25. Cryptojacking

Ciberataque donde el intruso utiliza el poder de procesamiento (CPU/GPU) de servidores Cloud víctimas para minar criptomonedas a expensas de facturas astronómicas.

26. IAM

Identity and Access Management: Capa fundamental que se encarga de gestionar quién puede acceder a qué recursos definiendo permisos exactos.

27. IaC

Infraestructura como Código (Terraform, AWS CloudFormation). Gestionar y aprovisionar toda la nube utilizando archivos de código legibles, en lugar de clicks manuales.

28. Shift-Left

Mover los controles de seguridad lo más al inicio del desarrollo, integrándolos desde el momento en que se escribe el código (Checkov, tfsec).

29. Tokens SAS Expuestos

Tokens (Shared Access Signatures) que al ser compartidos otorgan acceso delegado. Si se malconfiguran, exponen toda la información.
